# Bible Atlas — Data Layer Audit

**Status:** Work in progress. Sections are appended incrementally as investigation completes.

**Scope:** Read-only audit of the data layer (canon tables, verse counts, cross-reference integrity, Bible API, indices, preferences, liturgical data, genres, bundle impact). Findings from a prior audit session (31-book gap, 14-book gap, 60s error cache, `handleShowInbound` race) are deliberately excluded from this document.

**Severity legend:**
- **Critical** — silent core-flow failure
- **High** — user-visible non-core failure OR data inconsistency
- **Medium** — edge cases, load issues
- **Low** — technical debt
- **Informational** — FYI only

## 1. Canon and Book Coverage

`src/data/books.ts` defines **80 books** (66 Protestant + 7 Catholic deuterocanon + 4 Orthodox additions + 3 Ethiopian additions). Every other data/lookup table in the project covers only the 66-book Protestant canon, so any selection of a DC book resolves to a phantom — the book card renders, the chapter grid renders (from `book.chapters`), but every downstream lookup silently returns empty. The header comment in `chapter-verses.ts:1–3` explicitly notes it is Protestant-only; the other files don't advertise the limitation. `search-index.ts` (lines 82–100) *does* ship aliases for all 80 canonical names, which means a user can successfully search for "Tobit" or "1 Enoch" and have the palette happily route into a dead data pipeline.

## 3. Cross-reference Data Integrity

`arc-crossrefs.json` carries 327,880 arcs; `book-edges.json` carries 3,431 pre-aggregated book-to-book pairs (weights 3–10, no self-pairs). Every book pair in `book-edges.json` appears in `arc-crossrefs.json`, but the arc file also contains 519 same-book pairs that were correctly dropped during aggregation — they're within-book cross-references. Weight ranges look fine at the book level. At the verse level there are three real integrity problems: verse-level self-references, duplicate verse-pair arcs, and a mislabelled `weight` field that looks like a ranking signal but is just the source book's genre index (0–9).

## 4. Bible API Integration

`bible-api.ts` wraps every verse/passage lookup through the `/api/verse` server route (`route.ts`), which proxies bible-api.com and annotates responses with a 60-second cache revalidation. The module keeps a process-lifetime `Map` cache for both single-verse and passage fetches (null results included, so failed lookups are remembered). Fallback to the WEB translation when the requested translation fails is implemented for both paths. Coverage is 66 Protestant books only (see §1.2). Three real concerns: the static-export mismatch, a bug in `parseReadingReference` for comma-only lists without a dash, and a translation-fallback flag that's lost on the successful path.

## 5. Verse Index and Search Index

`verse-index.ts` fetches `/arc-crossrefs.json` once (promise-memoized via `indexPromise`) and builds a `Map<string, Reference[]>` keyed by `"{BOOK}-{chapter}-{verse}"`. `search-index.ts` pre-computes its own in-process index from `books.ts` at module load, tiered by exact/prefix/alias/fuzzy match. The two indices are independent — search never calls verse-index. Failure modes are mostly handled gracefully (book lookups return `[]` when unbuilt, fuzzy search skips single-char queries). The biggest concrete risk is the palette→fetch compound: `parseQuery` yields a `ParsedQuery` with a book + chapter + verse, but never validates that the (book, chapter, verse) tuple is in-range, so the palette can navigate to unreachable addresses that then silently fail in the reader.

## 6. Preferences and Persistence

`preferences.ts` wraps `localStorage` with SSR guards, JSON (de)serialization, a `QuotaExceededError` in-memory fallback, and a debounced `recent-passages` flusher. The 11 enumerated `PreferenceKey`s are each used at least once across `page.tsx`, `ReadingPlanCard.tsx`, `DetailPanel.tsx`, and `SearchPalette.tsx` — no dead keys. `clearPreferences` covers every key. SSR guards are correct (every public fn short-circuits on `typeof window === 'undefined'`). One real bug: the `last-view-date` 14-day staleness check uses UTC-parsed dates but `toISOString().slice(0,10)` for the write, which skews the boundary by up to ±1 day depending on the user's timezone. Otherwise healthy.

## 7. Liturgical and Readings

`liturgical.ts` computes Easter via the Anonymous Gregorian algorithm (correct), derives Ash Wednesday/Pentecost/Advent Sunday from it, and branches a six-way season string. `readings.ts` returns a hand-curated reading set per season and a 7-entry rotation for ordinary time indexed by `dayOfYear % 7`. The shape is defensible as a placeholder, but there are four real problems: (a) `'pentecost'` is a `LiturgicalSeason` but has no dedicated reading branch, so Pentecost Sunday falls through to Ordinary Time; (b) Lent reads are indexed by `dayOfWeek` instead of `(weekOfLent, dayOfWeek)`, so every Monday of Lent ships the same Gospel; (c) two readings reference deuterocanonical books (`Wisdom 2:1-22`, `Sirach 27:4-7`) that `bible-api.com` cannot fetch under any translation — both return `null` silently; (d) `parseReadingReference`'s comma-list handling returns the last numeric in the string as `endVerse`, which inflates the range used for UI ("Psalm 104:1-2, 5-6, 10, 12, 24, 35" becomes 1–35 for any downstream consumer that reads `endVerse`).

## 9. Data Loading and Bundle Impact

## Meta Observations

- **"Protestant canon is the real schema" assumption runs deep.** Every derived data structure — chapter-verses, bible-api names, book-edges, arc-crossrefs, per-book crossrefs, mcheyne streams — bakes in 66 books, while `books.ts` advertises 80. The UI wires up canon toggles that visually add/remove books, but nothing behind the toggle actually has DC data. Future sessions should treat `books.ts` as the product surface and treat every other file as a 66-book derived index. Adding a new canon today quietly breaks at every layer.
- **Verse counts are duplicated in three places** (`books.ts.verses`, sum of `chapter-verses.ts[id]`, `arc-crossrefs.json.books[].verses`) with no runtime assertion that they agree. The Genesis bug slipped past every layer because nothing cross-checks them. A one-time build script or a unit test that asserts the three agree would prevent this class of bug.
- **Two separate cross-reference datasets serve different purposes** — `arc-crossrefs.json` (verse-level, for rendering + margin dots) and `book-edges.json` (book-level, pre-aggregated, for Tooltip and ForceGraph). They're derived from the same source but the aggregation drops same-book pairs and re-buckets weights (3–10 clamped). The asymmetry is undocumented; future sessions may accidentally feed one where the other is expected.
- **Module-level caches are load-bearing for performance.** `verseCache`/`passageCache` in `bible-api.ts`, `verseTextCache`/`verseTextPending` in `search-index.ts`, `crossrefCache` in `crossref-utils.ts`, `cachedIndex`/`indexPromise` in `verse-index.ts`, `recentPassagesCache` in `preferences.ts`. Seven caches, none with TTL or size limits. This is fine for a page-lifetime SPA but becomes a leak if the app ever runs as a long-lived web view.
- **`Reference.weight` is dead data** (§3.3) — stored but never read. A signal that the data-shape evolved faster than the consumers. Worth an audit pass on other `types.ts` fields (`CrossRefEdge.weight` vs `.count`, `ChapterCrossRefSummary.topTargetBooks`) to confirm which fields are actually consumed.
- **`parseReadingReference`'s comma handling** is a small but notable source of "it happens to work today" fragility. If any component ever starts using `endVerse` as a tight upper bound (e.g. to scroll-lock the reader to the liturgical reading), the ordinary-time entries with comma lists will over-shoot.
